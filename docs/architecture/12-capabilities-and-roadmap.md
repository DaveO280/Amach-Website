The honest headline: **the hard, differentiating infrastructure is built and live.** Wallet-derived encryption, a real multi-agent pipeline, and a working on-chain attestation loop is a rare combination. The weaknesses are almost all in _cohesion_ (three hand-copies of everything) and _finish_ (features stopped at 80%), not in the core bets.

## 3. Strategic gaps — decisions, not tasks

1. **Encryption posture is trust-the-server, and nothing says so out loud.** All three clients POST the PBKDF2-derived AES key in every Storj request body; encryption/decryption happens server-side. That's a defensible design (enables server-side merge, quality scoring, AI context) — but for a data-sovereignty product it's a _positioning_ decision that should be explicit: either own it in the whitepaper/docs, or invest in moving crypto client-side (large: breaks server merge). Deciding this shapes R1 and the marketing story.
2. **There is no shared contract layer between the three clients.** The frozen signature message, PBKDF2 params, Storj payload schemas, timeline event shape, and Luma persona prompt each exist in 2–3 hand-synced copies. Every cross-platform bug in the audit traces back to this. R1 is the cheapest insurance available.
3. **Two AI chat brains diverge.** Web (`aiStore → LumaAiService → /api/venice`) and iOS (`ChatService → /api/ai/chat`) have separate personas, context assembly, and memory extraction. Users will eventually notice Luma has two personalities. Converging on one server-side chat route with per-client context injection is the structural fix.
4. **CLAUDE.md misleads the tooling.** Wrong agent count, renamed services, wrong deployed contract. Cheap to fix, high compounding value since every AI session reads it first.

## 5. What NOT to do

- **Don't rewrite the multi-agent system** — it works, it's cached, and its issues are duplication/logging, not architecture.
- **Don't break the thin-client principle** by letting iOS talk to Storj/Venice directly, even when convenient.
- **Don't bump OpenZeppelin** past the 5.4.0 pin without a storage-layout-validated upgrade plan (see [11](11-consolidation-plan.md) Phase 3 notes).
- **Don't add a fourth copy of anything** — new shared logic goes in the contract kit (R1) or the server, full stop.
